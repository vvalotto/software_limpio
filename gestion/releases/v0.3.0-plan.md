# Plan de Release v0.3.0 — ArchitectAnalyst

**Estado:** 🔄 Planificación
**Tipo:** GitHub Release
**Tag:** `v0.3.0`
**Alcance:** Implementación completa de ArchitectAnalyst — análisis estratégico de arquitectura al fin de sprint

---

## Contexto

ArchitectAnalyst es el tercer y último agente de la trilogía. Analiza la **salud arquitectónica del sistema completo** al finalizar un sprint, con foco en métricas estructurales y su evolución histórica. A diferencia de CodeGuard y DesignReviewer, **nunca bloquea** — es un agente informativo y estratégico.

### Decisiones de diseño acordadas

- **Scope:** sistema completo (no solo el delta del PR)
- **Trigger:** manual, fin de sprint o milestone
- **Métricas:** arquitectura pura — Métricas de Martin + ciclos de dependencias + violaciones de capas
- **Fuera de scope:** CVEs, coverage, duplicación, MI, tech debt (son salud general, no arquitectura)
- **Herramientas:** AST propio, sin dependencias nuevas
- **Modelo de datos:** tipos propios (`ArchitectureResult`, `ArchitectureSeverity`) — comparte solo `Verifiable` de `shared/`
- **IA:** opt-in (el agente funciona completo sin ella; con IA agrega interpretación estratégica)
- **Output:** terminal Rich + JSON (sin dashboard HTML ni Plotly por ahora)
- **Exit code:** siempre 0 (nunca bloquea; informativo)
- **Tendencias:** histórico en SQLite, comparación sprint a sprint con indicadores ↑ ↓ =

### Métricas a implementar

| # | Métrica | Descripción | Umbral | Severidad | Herramienta |
|---|---------|-------------|--------|-----------|-------------|
| 1 | Ca (Afferent Coupling) | Módulos externos que dependen de este módulo | — (referencia) | INFO | AST propio |
| 2 | Ce (Efferent Coupling) | Módulos de los que depende este módulo | — (referencia) | INFO | AST propio |
| 3 | I (Instability) | Ce / (Ca + Ce) | ≤ 0.8 | WARNING | Cálculo |
| 4 | A (Abstractness) | Clases abstractas / total clases del módulo | — (referencia) | INFO | AST propio |
| 5 | D (Distance) | \|A + I - 1\| — distancia a la Main Sequence | ≤ 0.3 WARNING / ≤ 0.5 CRITICAL | WARNING/CRITICAL | Cálculo |
| 6 | Ciclos de dependencias | Ciclos en el grafo de imports del proyecto | = 0 | CRITICAL | AST propio + DFS |
| 7 | Violaciones de capas | Imports que violan la arquitectura declarada en config | = 0 | CRITICAL | AST propio + config |

## Fase 2 — Métricas de Martin

**Objetivo:** Implementar las 5 métricas de Robert C. Martin que caracterizan la salud arquitectónica de cada módulo.

- [ ] **2.1** `DependencyGraphBuilder` — helper interno que construye el grafo dirigido de imports de todo el proyecto via AST (base compartida para Martin y ciclos)
- [ ] **2.2** `CouplingAnalyzer` — calcula Ca y Ce por módulo usando el grafo de dependencias
- [ ] **2.3** `InstabilityAnalyzer` — calcula I = Ce / (Ca + Ce) por módulo; detecta módulos extremadamente inestables (I > 0.8)
- [ ] **2.4** `AbstractnessAnalyzer` — calcula A = clases abstractas / total clases por módulo via AST
- [ ] **2.5** `DistanceAnalyzer` — calcula D = |A + I - 1| por módulo; clasifica en Zone of Pain (A≈0, I≈0) y Zone of Uselessness (A≈1, I≈1)
- [ ] **2.6** Tests unitarios para cada analyzer (con fixtures de código Python sintético)

**Criterio de cierre:** Los 5 analyzers calculan correctamente sobre código de prueba con módulos en distintas posiciones del Main Sequence.

---

## Fase 3 — Análisis Estructural

**Objetivo:** Detectar los dos problemas arquitectónicos más críticos: ciclos de dependencias y violaciones de la arquitectura en capas.

- [ ] **3.1** `DependencyCyclesAnalyzer` — detecta ciclos en el grafo de imports del proyecto (reutiliza `DependencyGraphBuilder`, DFS propio sin dependencias nuevas)
- [ ] **3.2** `LayerViolationsAnalyzer` — valida imports entre módulos contra las reglas de capas declaradas en config; detecta imports que violan la dirección permitida
- [ ] **3.3** Tests unitarios para cada analyzer

**Criterio de cierre:** Ambos analyzers detectan violaciones correctamente en código de prueba con problemas intencionales.

---

## Fase 4 — Tendencias Históricas

**Objetivo:** Persistir snapshots de cada análisis en SQLite y calcular la evolución de métricas entre sprints.

- [ ] **4.1** Refactorizar `SnapshotStore` (ya existe en skeleton) — guardar y recuperar snapshots con `sprint_id` opcional y timestamp
- [ ] **4.2** Implementar `TrendCalculator` — compara snapshot actual vs anterior; calcula delta y clasifica en ↑ (empeora) ↓ (mejora) = (estable) por métrica
- [ ] **4.3** Integrar trends en `ArchitectureResult` — cada resultado incluye su indicador de tendencia cuando hay histórico disponible
- [ ] **4.4** Tests unitarios de `SnapshotStore` y `TrendCalculator`

**Criterio de cierre:** Dos análisis consecutivos muestran tendencias correctas en el output.
